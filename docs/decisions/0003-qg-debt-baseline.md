# ADR 0003 — QG debt baseline: the engine gates itself (E0.6)

**Status:** Accepted · **Date:** 2026-07-06
**Relates to:** E0.4 (baseline & ratchet mechanism), E0.2 (self-CI), E0.5
(self-exemption — shrinks this baseline), E0.6 (this ADR)

## Context

The engine shipped a Clean-as-You-Code ratchet (E0.4) and a self-CI
workflow (E0.2), but its own scan stayed report-only: a full audit found
hundreds of findings, most of them legacy debt or the scanner matching its
own rule-definition strings. A mass remediation would be a huge,
high-risk diff; staying report-only means the gate never gates its owner.

E0.6 takes the third path: **commit the debt as a reviewed, per-file
baseline and flip self-CI to blocking**. Tolerated debt passes; any
regression, any new sub-floor file, and any security veto fails CI.

## Decision

**1. The committed `.quality-gate.baseline.json` is generated by the E0.4
CLI (`python quality-gate/quality_gate.py --root . --mode baseline`) at
the E0.6 merge state.** Identity is per-file counts
(`{prs, errors, warnings, vetoed}`), not finding fingerprints — the v1
scheme E0.4 shipped; a findings-hash identity remains the documented v2
upgrade path. The exact totals derived from the committed artifact
(kept honest by `quality-gate/tests/test_self_baseline.py`, which parses
both this line and the baseline):

`baseline-totals: files=177 errors=147 warnings=1292 below_floor=76 vetoed=0`

**2. Error-severity findings inside baselined, non-regressed files no
longer fail `--mode check`** (they stay in the report and the stats as
`baseline_tolerated_errors`). Without this, any baselined repo carrying
error-severity debt could never go green and the ratchet would be
unusable — toleration is exactly what a baseline is for. A regression or
a veto removes the file from the tolerated set, so every NEW error still
blocks. (Shipped with a related one-line fix: pack findings emitted with
`qg.types.Severity` enums bypassed PRS counting entirely; severities are
now normalized at the engine bridge, which is why the totals above are
slightly higher than the raw pre-fix audit.)

**3. Real debt vs. self-scan false positives.** The 147 baselined errors
split three ways; the FP share must NOT be "fixed" by editing detection
pattern strings (obfuscating patterns to dodge your own scanner is
dishonest) — it expires with E0.5's self-exemption layer:

| Bucket | Errors | Dominant rules | Nature | Expiry |
|---|---|---|---|---|
| Engine/product code (real debt) | 72 | function_size 47, sql_string_interpolation 10, no_silent_catch 6, db_call_in_loop 5 | Genuine oversized functions and quality debt | Fix incrementally under the ratchet |
| `quality-gate/qg/*` rule packs (self-scan FPs + real size debt) | 54 | function_size 37, unbounded_graph_execution 5, LLM-PY-UNVALIDATED-JSON 3 | Rule pattern strings flagging themselves — e.g. `checks_security.py:356` (its own secret regex), `checks_databases.py:24` (its redis URI regex), `checks_debug.py:85-131` (its own debug-token patterns, warnings) — plus real function_size debt in pack code | FP share **expires with E0.5**; function_size debt fixed incrementally |
| `quality-gate/tests/*` fixture files (self-scan FPs) | 21 | missing_await 8, LOOP-PY-NO-MAX-ITERATIONS 5, LLM-PY-UNVALIDATED-JSON 3 | Test fixtures intentionally containing the bad patterns the rules detect | **Expires with E0.5** (test-fixture exemption) |

Also tagged as an FP expiring with E0.5: `fix-engine/fe/llm_fixes.py:39`
(`env_variable_exposure` fires on `return bool(os.environ.get(...))`,
which returns a bool, not the environment).

**4. Vetoes are not baselineable — so none were baselined.** Four files
were PRS-vetoed by their own docstring/fixture strings pairing veto-rule
tokens with LLM-call tokens (`qg/checks_llm_output_safety.py` docstring;
`tests/test_agent_loop_safety.py`, `tests/test_agentic_harness.py`,
`tests/test_llm_output_safety.py` fixtures). Per E0.4 doctrine a baseline
never masks a veto, so these could never go green. The docstring was
reworded and the fixtures are now assembled at runtime (the sanctioned
pattern from `test_baseline_ratchet.py`) — **no detection pattern string
was altered**, and the fixtures the tests write to disk are unchanged.
`vetoed=0` in the totals line is asserted by the self-test.

**5. Ratchet direction is proven in the same PR** by three cheap real
fixes: `reporting/__init__.py` 84.0 → ≥85 (dead variable removed,
`is_file()` probes hoisted into one `_report_entry` helper);
`fix-engine/fe/fixes_additional.py` 3 → 1 errors (generated-comment
marker tokens reworded to neutral, issue-free forms; two dead `indent`
variables removed); `cathedral-keeper/tests/test_coherence.py` 1 → 0
errors (marker-token section comment reworded). The remaining two files
stay below the floor (function_size debt and interface-required
`config` parameters) — deliberately left to the ratchet, not
mass-remediated here.

**6. `quality_gate.py` itself (the monolith, 1400+ lines, PRS 0.0) is
baselined, not split.** The split is real work with real regression risk
and gets its own backlog item; baselining it now unblocks blocking
self-CI today while the ratchet guarantees it cannot get worse.

**7. CK alignment: one floor, one baseline, low-severity ingestion.**
CK's QG ingestion (`cathedral_keeper/integrations/quality_gate.py`) reads
the same committed baseline; findings for baselined, non-regressed,
non-vetoed files ingest at **low** severity instead of high, so
`ck analyze` exits 0 on tolerated debt and CK_BLOCKING flipped to True
(the workflow's last `continue-on-error` mask is gone). A missing or
corrupt baseline fails **open to high** — a broken baseline makes CK
stricter, never quieter. The ingestion cut-off is pinned to **85**
(`PRS_FLOOR`), the same floor QG enforces — the historical ~90-vs-85
ambiguity is resolved to one number.

**8. Self-CI is blocking.** `ENGINE_PROFILE["QG_CMD"]` is
`--root . --mode check --baseline .quality-gate.baseline.json --json
--sarif qg.sarif`; `CK_BLOCKING=True`; `make check` runs the identical
QG command locally.

## Consequences

- Any PR that regresses a baselined file (more errors, more warnings, or
  a lower PRS) fails CI with a `baseline_ratchet` error naming the
  metric; new files face the full PRS-85 floor; vetoes always fail.
- The baseline shrinks only deliberately: regenerating it requires a
  local `--mode baseline` run (refused under CI env) and a reviewed diff
  of a protected-surface file (CODEOWNERS).
- E0.5 (self-exemption) should mechanically shrink the FP share of this
  baseline (buckets 2 and 3 above); the totals line and the coupled
  self-test must be regenerated together in that PR.
- The count-based identity can, in principle, let one new error replace
  one fixed error inside the same file unnoticed (documented E0.4 v1
  trade-off); the findings-hash v2 identity closes that gap.
